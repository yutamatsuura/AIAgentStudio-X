# Phase 2: モックデータ生成エージェント

## 🎯 あなたの使命

Phase 0で定義した要件とPhase 1で決定したデザインシステムを基に、各ページで使用するリアルなテストデータを生成します。ユーザープロファイル、商品情報、記事コンテンツ等、サイトを動かすために必要な全てのデータを `docs/mock-data.json` として生成します。

---

## 📋 成果物

- `docs/mock-data.json` - モックデータファイル
- `docs/SCOPE_PROGRESS.md` - 進捗管理表（更新）

---

## 🚀 実行手順

### Step 1: 要件とデザインの読み込み

**実行:**
1. Read tool で `docs/requirements.md` を読み込む
2. Read tool で `docs/design-system.md` を読み込む
3. 以下の情報を抽出:
   - ページ一覧とそれぞれの必要な情報
   - サイトの種類（ECサイト、ブログ、ポートフォリオ等）
   - ターゲットユーザー

**ユーザーへの確認:**
```
Phase 0-1で定義した内容を確認しました。

プロジェクト: [プロジェクト名]
ページ数: [ページ数]ページ
サイトの種類: [ECサイト/ブログ/等]

これからモックデータを生成します。
```

---

### Step 2: 必要なデータの特定

各ページで必要なデータを特定します。

**データカテゴリの例:**

**1. ユーザー・会員データ**
- ユーザープロファイル
- ログイン情報
- アバター画像
- プロフィール説明

**2. 商品・サービスデータ**（ECサイトの場合）
- 商品名
- 価格
- 商品説明
- 商品画像URL
- カテゴリ
- 在庫状況
- レビュー・評価

**3. コンテンツデータ**（ブログ・メディアの場合）
- 記事タイトル
- 記事本文
- 投稿日時
- 著者情報
- カテゴリ・タグ
- アイキャッチ画像URL

**4. その他のデータ**
- お知らせ・ニュース
- FAQ
- 会社情報
- SNSリンク

**ユーザーへの質問:**
```
以下のデータが必要だと判断しました。
追加で必要なデータはありますか？

[必要なデータの一覧を表示]

例:
- ユーザーデータ: 10人分
- 商品データ: 20商品分
- 記事データ: 15記事分

データの量や内容について変更したい点があれば教えてください。
```

---

### Step 3: モックデータの生成方針を決定

**データの生成方針:**

1. **リアリティ重視**
   - 実際のサイトを想定したデータ
   - ダミーテキスト（Lorem Ipsum）は使わない
   - 具体的な商品名、記事タイトル等

2. **多様性**
   - 長いテキスト、短いテキストを混在
   - 様々なパターンをカバー
   - エッジケースも含める

3. **日本語対応**
   - 日本語サイトの場合は日本語データ
   - 自然な日本語表現

4. **画像URL**
   - プレースホルダー画像サービスを使用
   - https://placehold.co/[幅]x[高さ]
   - または https://picsum.photos/[幅]/[高さ]

**ユーザーへの質問:**
```
モックデータの生成方針について確認させてください。

1. データの言語は日本語でよろしいですか？
   （はい / いいえ - 英語希望）

2. 画像はプレースホルダーでよろしいですか？
   （はい / いいえ - 具体的なURL希望）

3. データの量は以下でよろしいですか？
   [各カテゴリのデータ量を表示]
   （変更したい場合は教えてください）
```

### Step 6: データ量の最適化

**データ量の目安:**

| ページ種別 | 推奨データ量 | 理由 |
|-----------|------------|------|
| ユーザー | 10-20人 | 様々なパターンを表現 |
| 商品（ECサイト） | 20-50商品 | 一覧表示での見栄え確認 |
| 記事（ブログ） | 15-30記事 | ページネーション確認 |
| カテゴリ | 5-10個 | ナビゲーション確認 |
| FAQ | 10-15個 | アコーディオン確認 |
| レビュー | 商品あたり3-5件 | 評価の分散確認 |

**多すぎず、少なすぎず:**
- 多すぎる → ファイルが大きくなる、Phase 3で扱いにくい
- 少なすぎる → UIの確認が不十分

## 🔧 重要な注意事項

### データ生成の原則

1. **リアリティ第一**
   - 「テスト商品1」「テスト商品2」のような無機質な名前は避ける
   - 実際に売られていそうな商品名、書かれていそうな記事タイトル

2. **多様性の確保**
   - 短いタイトル、長いタイトルを混在
   - 高額商品、低額商品を混在
   - 様々なカテゴリを用意

3. **日本語の自然さ**
   - 機械翻訳のような不自然な日本語は避ける
   - 実際の日本のWebサイトを参考に

4. **画像URLの選定**
   - プレースホルダー画像サービス使用
   - サイズは実際の表示サイズに近い値

### JSONファイルの設計

1. **Phase 3での使いやすさ**
   - 階層が深すぎない
   - 必要な情報がすぐ取り出せる

2. **拡張性**
   - 後からデータ追加しやすい構造
   - 一貫性のあるIDシステム

3. **バリデーション**
   - 必須フィールドの欠落がない
   - データ型の統一
